### индексирование и поиск по индексу
 схема индексирования
 1. Item содержит поле search_content в котором собраны все фразы из всех текстовых полей этой и связнных таблиц
 2. сервис Clickhouse содержит связанную таблицу wine_replica.items с двумя полями id и search_content - это реплика Item
 3. схема в clickhouse wine_replica.items -> deafult.items_search_mv -> default.items_search
 4. эта таблица имеет индекс словарный индекс по которму происходит поиск в два этапа:
    1. отфильтровывает записи содержащие любое из слов/токенов поискового запрооса (экстремально быстро);
    2. нечеткий поиск с опечатками (допускается одна опечатка в слове/токене) - медленнее, но поиск идет уже по отфильтрованному списку
    3. там же определеяется RANGE и результат ранжируется по релевантности
 5. Результат передается в fastapi -> postgresql и там делается выборка по ids с ограничением 20 записей (наиболее релевантных) - это тоже очень быстрый запрос
 6. до 1 млн записей чистый полнотекстовый индекс postgresql работает быстрее, затем clickhouse обгоняет

   1. (clickhouse реплицирует записи моментально - заполнение после запси (patch) - это сложная операция в clickhouse)
      1. добавление  1 варианта с пустым search_content -> возникает конфликт в clickhouse. для него это источник для индекса, репликация может зависнуть
      2. update - добавление 2 варианта с версией 2
      3. получается две записи с одним id и двумя версиями - требуется применение оператора FINAL для фильтарции от старых версий
